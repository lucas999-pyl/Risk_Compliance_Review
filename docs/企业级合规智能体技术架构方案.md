# 化工合规 RAG 工具 · 技术方案

## 1. 方案定位

本方案聚焦化工物料合规预审，不做泛行业智能体包装。系统目标是把官方优先知识库、供应商资料包、函数预检、RAG 检索、规则引擎、必要 Agent 分工、主审汇总和人工复核边界连接成一条可审计的预审链路。

主链路如下：

```text
上传官方/内部知识库源文档 -> 生成 Chunk 和向量库 -> 上传 SDS/配方/工艺资料
-> 勾选标准检查项 -> 函数预检资料完整性 -> 多 query RAG -> 规则命中
-> 物料/工艺/储运/法规 Agent 分支分析 -> 主审三值判定 -> 结构化整改报告
```

系统定位为 AI 辅助预审，不替代最终法律、法规或 EHS 合规意见。

## 2. 真实业务难点

化工合规审查不是普通问答，核心难点包括：

- 企业用户需要标准检查项，不应每次手写自然语言任务。
- 资料来源复杂：SDS、配方表、工艺参数、储运条件、供应商声明格式不统一。
- 物质识别敏感：CAS、EC、中文名、英文名、别名、保密成分和未知 CAS 都会影响判断。
- 法规与企业规则并存：OSHA HCS、ECHA SVHC、EPA TSCA、中国危险化学品目录和企业内部红线不能混为一谈。
- 确定性规则必须优先：章节完整性、字段缺失、禁忌组合、未知物质、企业红线不能交给大模型自由发挥。
- 证据必须可追溯：每个结论需要来源、版本、生效日期、retrieved_at、source_url、规则原文和用户原文。
- 复核边界必须保守：知识库未加载、资料缺失、无法匹配、子 Agent 冲突时必须进入复核。

## 3. 函数优先，不是 Chatbot

Chatbot 可以解释概念，但不适合承载合规预审责任边界。本方案采用“函数预检 + 受控 RAG Agent”架构：

- 函数预检负责 SDS/配方/工艺资料齐套性、章节、字段和关键词存在性。
- RAG 负责召回官方/内部知识证据。
- 规则引擎负责硬性红线、阈值、禁忌矩阵和保守复核策略。
- Agent 只保留在需要上下文确认、组合关系和冲突推理的环节。
- 主审节点按规则优先原则汇总，不允许 LLM 覆盖硬规则。
- Trace、chunk、Rerank 和证据链可回放，用户端报告只呈现可整改结论。

## 4. 总体架构

```text
知识库源文档层
OSHA HCS/SDS、ECHA SVHC、EPA TSCA、应急管理部危化目录、企业内部禁忌矩阵
        |
        v
知识库构建层
manifest 校验、源文档解析、chunk 切分、metadata 绑定、embedding、SQLite 向量索引
        |
        v
资料输入层
SDS、配方表、工艺说明、目标市场、标准检查项 enum
        |
        v
函数预检层
文件齐套、SDS 章节、CAS/浓度、工艺温压、关键步骤、储存条件、补件动作
        |
        v
RAG 检索层
任务拆解、多 query、向量召回、keyword/CAS exact match、rules rerank、TopK 证据
        |
        v
受控 Agent 判断层
物料 Agent、工艺 Agent、储运 Agent、法规 Agent
        |
        v
主审与报告层
交叉质检、规则优先、三值判定、结构化整改报告、管理端技术 trace
```

## 5. 官方优先知识库

主知识库来自可审计的源文档包：

```text
data_samples/chemical_knowledge_sources/official_pack_2026_05/
```

每份源文档包含：

- `title`
- `jurisdiction`
- `source_type`
- `source_origin`
- `quality_tier`
- `document_role`
- `source_url`
- `version`
- `effective_date`
- `retrieved_at`
- `license_note`
- 正文摘要和规则化条款

官方优先来源：

- OSHA HCS 2024 / SDS / Appendix D
- ECHA Candidate List / SVHC
- EPA TSCA Inventory
- 应急管理部《危险化学品目录（2015版）》实施指南
- 企业内部禁忌矩阵，明确标记为 `internal_policy`

`chemical_rules_pack.json` 仅保留为开发备用包，不作为主知识库来源。

## 6. 固定检查项入口

主接口：

```text
POST /chemical/upload-review
```

用户端输入：

- `check_types`: `material`, `process`, `storage`, `regulatory`
- `target_markets`
- `top_k`
- `sds_file`
- `formula_file`
- `process_file`

`review_task` 仅保留为兼容和管理端高级调试字段。用户端提交 `check_types` 后，系统生成固定任务描述：

```text
执行固定检查项：物料合规、工艺适用性、储运合规、法规初筛。
```

## 7. Agent 责任边界

### 资料完整性函数预检

核对 SDS、配方表、工艺说明是否足以支撑预审。缺 SDS 章节、修订日期、供应商、CAS、浓度、工艺温压、关键步骤、储存条件时，生成 `document_quality`、`supplement_actions` 和结构化报告中的资料完整性复核项。

### 物料 Agent

处理链路是：提取物质名/CAS/浓度 -> 生成物料组合 -> 命中物质主数据、禁忌矩阵或企业红线 -> 回到用户原文确认是否实际共存或同用 -> 输出三值判断。

### 工艺 Agent

处理链路是：提取温度、压力、关键步骤、设备和用途 -> 匹配工艺风险规则 -> 判断是否存在同釜混配、高温氧化剂、酸化次氯酸盐等操作 -> 输出三值判断。

### 储运 Agent

处理链路是：提取储存类别、储存条件、UN 编号、运输信息和隔离要求 -> 匹配储运兼容性规则 -> 判断禁忌物是否同储、运输信息是否冲突、可燃品是否缺防火要求 -> 输出三值判断。

### 法规 Agent

结合目标市场和 RAG 证据，进行危险化学品目录、SVHC、TSCA、OSHA HCS 等初筛。知识库未加载或未召回版本化证据时，不允许自动放行。

### 主审 Agent

按规则优先原则汇总各分支：

- `合规`：资料完整、知识库已加载、证据充分、未命中红线。
- `复核`：资料缺失、未知物质、知识库未加载、证据不足或分支冲突。
- `不合规`：命中硬性拦截规则，例如禁忌组合或企业红线。

## 8. 结构化整改报告

用户端报告只展示 `不合规` 和 `复核` 项，合规项不展开。每条问题包含：

- 编号
- 维度：资料完整性、物料、工艺、储运、法规
- 违反规则编号
- 规则原文或规则解释
- 用户原文
- 改进建议

内部检索详情、命中 chunk、Rerank 排序、Agent 分支和 Trace JSON 保留在管理端或技术折叠区。

## 9. 输出契约

上传审查响应的关键字段：

- `check_types`
- `review_task`

## 10. 边界声明

当前系统仍需在生产化前补齐完整 OCR、法规授权来源、持续更新机制、权限、多租户、审批流和企业系统集成。官方来源文本用于构建可检索知识文档，真实业务上线前需要合法授权、持续更新、人工审核和客户 EHS/法规团队确认。
