# 化工合规 RAG 工具 · 技术方案

## 1. 方案定位

本方案聚焦化工物料合规预审，不做泛行业智能体包装。系统目标是把官方优先知识库、供应商资料包、RAG 检索、规则引擎、多 Agent 分工、主审汇总和人工复核边界连接成一条可审计的预审链路。

主链路如下：

```text
上传官方/内部知识库源文档 -> 生成 Chunk 和向量库 -> 上传审查项目资料包
-> 资料包软门禁预检 -> 选择审查场景和业务检查项 -> 资料完整性预检 -> 任务路由 -> 多 query RAG
-> 规则命中 -> 多 Agent 分支分析 -> 主审三值判定 -> 客户报告 / 管理端技术 trace
```

系统定位为 AI 辅助预审，不替代最终法律、法规或 EHS 合规意见。

## 2. 真实业务难点

化工合规审查不是普通问答，核心难点包括：

- 资料来源复杂：SDS、配方表、工艺参数、储运条件、供应商声明格式不统一。
- 物质识别敏感：CAS、EC、中文名、英文名、别名、保密成分和未知 CAS 都会影响判断。
- 法规与企业规则并存：OSHA HCS、ECHA SVHC、EPA TSCA、中国危险化学品目录和企业内部红线不能混为一谈。
- 确定性规则必须优先：禁忌组合、资料缺失、未知物质、企业红线不能交给大模型自由发挥。
- 证据必须可追溯：每个结论需要来源、版本、生效日期、retrieved_at、source_url 和 chunk 证据。
- 复核边界必须保守：知识库未加载、资料缺失、无法匹配、子 Agent 冲突时必须进入复核。

## 3. 为什么不是 Chatbot

Chatbot 可以解释概念，但不适合承载合规预审责任边界。本方案采用受控 RAG Agent 架构：

- RAG 负责召回官方/内部知识证据。
- 规则引擎负责硬性红线、阈值、禁忌矩阵和保守复核策略。
- 确定性函数优先处理资料完整性、字段缺失、硬规则和可解释预检。
- 多 Agent 分别处理物料、工艺、储运、法规，以及需要语义判断的资料充分性问题。
- 主审节点按规则优先原则汇总，不允许 LLM 覆盖硬规则。
- Trace、chunk、Rerank 和证据链可回放，便于验证系统不是预设结论。

## 4. 总体架构

```text
知识库源文档层
OSHA HCS/SDS、ECHA SVHC、EPA TSCA、应急管理部危化目录、企业内部禁忌矩阵
        |
        v

manifest 校验、源文档解析、chunk 切分、metadata 绑定、embedding、SQLite 向量索引
        |
        v
资料输入层
审查项目、单文件/多文件资料包、审查场景、固定检查项、目标市场
        |
        v

SDS 章节、CAS/EC、浓度、GHS H/P、UN 编号、修订日期、温度、压力、储存条件
        |
        v
```

## 5. 官方优先知识库

主知识库来自可审计的源文档包：

```text
data_samples/chemical_knowledge_sources/official_pack_2026_05/
```

每份源文档包含：

- `title`
- `jurisdiction`
- `source_type`
- `source_origin`
- `quality_tier`
- `document_role`
- `source_url`
- `version`
- `effective_date`
- `retrieved_at`
- `license_note`
- 正文摘要和规则化条款

官方优先来源：

- OSHA HCS 2024 / SDS / Appendix D
- ECHA Candidate List / SVHC
- EPA TSCA Inventory
- 应急管理部《危险化学品目录（2015版）》实施指南
- 企业内部禁忌矩阵，明确标记为 `internal_policy`

`chemical_rules_pack.json` 仅保留为开发备用包，不作为主知识库来源。

## 6. 客户级审查项目闭环

主接口：

```text
POST /chemical/upload-review
```

普通用户输入：

- `review_scenario`
- `check_types`
- `target_markets`
- `top_k`
- `documents`

`documents` 支持单文件或多文件资料包。系统不会要求用户严格按三件套或固定命名上传，而是先输出 `package_precheck`：

- 每个文件识别为 `sds`、`formula`、`process`、`storage_transport`、`regulatory_certificate`、`test_report` 或 `unknown`。
- 每个文件标记 `readability`、识别置信度、已识别字段、缺失字段和客户可读提示。
- 资料包整体给出 `available_checks`、`limited_checks`、`blocked_checks`、`missing_documents`、`supplement_actions` 和 `overall_status`。
- 文本型 PDF 和 TXT 可以直接解析；扫描 PDF 暂不 OCR，进入不可读补件动作。

预检采用软门禁：资料不完整时仍可继续初筛，但客户报告必须说明哪些结论可靠、哪些只能复核、哪些需要补件。

管理端兼容输入：

- `review_task`
- `sds_file`
- `formula_file`
- `process_file`

普通用户端不暴露自然语言 `review_task`。管理端保留该字段，用于内部测试、复现问题和排错。系统不会只用标题生成一个总 query，而是把 `review_scenario`、`check_types`、目标市场、资料包预检结果和资料内容路由成多个子任务。

一级场景包括供应商准入预审、替代物料评估、目标市场合规筛查、工艺导入风险评估、储运与现场安全评估。

二级业务检查项包括资料完整性与可审性、成分识别与 CAS/浓度完整性、禁限用物质与红线物质筛查、物料相容性与危险组合、SDS 关键章节核查、工艺条件适配性、储存与运输条件核查、目标市场法规匹配、供应商声明/检测报告一致性、人工复核与补件建议。

旧版 `document_completeness/material/process/storage/regulatory` 保持兼容，后端会映射到新版业务检查项。

只有用户选择的 `check_types` 会生成对应子任务。每个子任务生成独立 RAG query，并记录在 `rag_queries`、`retrieval.by_query` 和 `agent_branches` 中。

## 7. 多 Agent 分工

### 资料完整性预检

资料完整性优先使用确定性函数，不作为普通 Agent 分支优先展示。函数预检负责核对文件类型、SDS 章节、修订日期、供应商、CAS、浓度、工艺温压、关键步骤和储存条件。缺失项生成 `document_quality`、`supplement_actions`，并进入 `customer_report`。

当字段存在但内容不足、跨文件矛盾、或资料无法支撑所选审查目标时，再保留 LLM/Agent 做语义充分性复核。

### 物料 Agent

识别物质、CAS/EC、浓度、别名和主数据命中情况。未知 CAS、缺 CAS、保密成分缺少合规声明时进入复核。

### 工艺 Agent

检查温度、压力、关键步骤、设备和混配关系。可燃液体与氧化剂同釜混配等情形触发硬性拦截。

### 储运 Agent

检查储存类别、禁忌物隔离、UN 编号、运输信息和储运兼容性。储存条件缺失或禁忌组合进入复核或不合规。

### 法规 Agent

结合目标市场和 RAG 证据，进行危险化学品目录、SVHC、TSCA、OSHA HCS 等初筛。知识库未加载或未召回版本化证据时，不允许自动放行。

### 主审 Agent

按规则优先原则汇总各分支：

- `合规`：资料完整、知识库已加载、证据充分、未命中红线。
- `复核`：资料缺失、未知物质、知识库未加载、证据不足或分支冲突。
- `不合规`：命中硬性拦截规则，例如禁忌组合或企业红线。

## 8. 客户报告与资料完整性

资料完整性模块决定审查是否具备继续判断的资料基础。输出包括：

- 完整性评分
- 阻断性缺口
- 字段级清单
- 原文证据
- 影响范围
- 补件动作
- 是否触发人工复核

这些结果进入 `review_workbench.document_quality`、`review_workbench.supplement_actions`、`chief_synthesis.review_items` 和 `customer_report`。

`customer_report` 是普通用户主报告，只展示不合格、复核和补件事项。报告先展示 `review_scope`，说明资料包状态、可审范围、受限检查项、阻断检查项和补件动作。每条问题固定包含：

- 编号
- 状态
- 规则编号
- 规则原文
- 用户原文
- 影响说明
- 整改建议

合规项只做汇总，不展开 RAG chunk、rerank、agent 分支或 trace 节点。

## 9. 可审计技术证据

系统保留可回放的技术证据，但这些内容进入管理端，不作为客户报告主视图：

- 知识 Chunk 和 vector_status
- query 和查询历史
- 多 query RAG 召回
- TopK 原始召回
- Rerank 后排序和原因
- 命中规则
- Agent 分支分析
- 主审汇总
- Trace JSON

## 10. 边界声明

当前系统仍需在生产化前补齐完整 OCR、法规授权来源、持续更新机制、权限、多租户、审批流和企业系统集成。官方来源文本用于构建可检索知识文档，真实业务上线前需要合法授权、持续更新、人工审核和客户 EHS/法规团队确认。
