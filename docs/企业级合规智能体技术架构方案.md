# 化工合规 RAG 工具 · 技术方案

## 1. 方案定位

本方案聚焦化工物料合规预审，不做泛行业智能体包装。系统目标是把官方优先知识库、供应商资料包、RAG 检索、规则引擎、多 Agent 分工、主审汇总和人工复核边界连接成一条可审计的预审链路。

Demo2.6 的主线是：

```text
上传官方知识库源文档 -> 生成 Chunk 和向量库 -> 输入审查任务 -> 上传 SDS/配方/工艺资料
-> 任务拆解 -> 多 query RAG -> 规则命中 -> 多 Agent 分支分析 -> 主审三值判定
```

系统定位为 AI 辅助预审，不替代最终法律、法规或 EHS 合规意见。

## 2. 真实业务难点

化工合规审查不是普通问答，核心难点包括：

- 资料来源复杂：SDS、配方表、工艺参数、储运条件、供应商声明格式不统一。
- 物质识别敏感：CAS、EC、中文名、英文名、别名、保密成分和未知 CAS 都会影响判断。
- 法规与企业规则并存：OSHA HCS、ECHA SVHC、EPA TSCA、中国危险化学品目录和企业内部红线不能混为一谈。
- 确定性规则必须优先：禁忌组合、资料缺失、未知物质、企业红线不能交给大模型自由发挥。
- 证据必须可追溯：每个结论需要来源、版本、生效日期、retrieved_at、source_url 和 chunk 证据。
- 复核边界必须保守：知识库未加载、资料缺失、无法匹配、子 Agent 冲突时必须进入复核。

## 3. 为什么不是 Chatbot

Chatbot 可以解释概念，但不适合承载合规预审责任边界。本方案采用受控 RAG Agent 架构：

- RAG 负责召回官方/内部知识证据。
- 规则引擎负责硬性红线、阈值、禁忌矩阵和保守复核策略。
- 多 Agent 分别处理资料完整性、物料、工艺、储运、法规问题。
- 主审节点按规则优先原则汇总，不允许 LLM 覆盖硬规则。
- Trace、chunk、Rerank 和证据链可回放，便于客户验证系统不是预设结论。

## 4. 总体架构

```text
知识库源文档层
OSHA HCS/SDS、ECHA SVHC、EPA TSCA、应急管理部危化目录、企业内部禁忌矩阵
        |
        v
知识库构建层
manifest 校验、源文档解析、chunk 切分、metadata 绑定、embedding、SQLite 向量索引
        |
        v
资料输入层
审查任务、SDS、配方表、工艺说明、目标市场
        |
        v
解析与标准化层
SDS 章节、CAS/EC、浓度、GHS H/P、UN 编号、修订日期、温度、压力、储存条件
        |
        v
RAG 检索层
任务拆解、多 query、向量召回、keyword/CAS exact match、rules rerank、TopK 证据
        |
        v
多 Agent 判断层
资料完整性 Agent、物料 Agent、工艺 Agent、储运 Agent、法规 Agent
        |
        v
主审与复核层
交叉质检、规则优先、三值判定、证据链、补件动作、预审报告摘要
```

## 5. 官方优先知识库

Demo2.6 不再把短小规则包作为客户主展示。主知识库来自手动上传的源文档包：

```text
data_samples/chemical_knowledge_sources/official_pack_2026_05/
```

每份源文档包含：

- `title`
- `jurisdiction`
- `source_type`
- `source_origin`
- `quality_tier`
- `document_role`
- `source_url`
- `version`
- `effective_date`
- `retrieved_at`
- `license_note`
- 正文摘要和规则化条款

官方优先来源：

- OSHA HCS 2024 / SDS / Appendix D
- ECHA Candidate List / SVHC
- EPA TSCA Inventory
- 应急管理部《危险化学品目录（2015版）》实施指南
- 企业内部禁忌矩阵，明确标记为 `internal_policy`

短小 `chemical_rules_pack.json` 只保留为开发备用包，不作为客户演示主入口。

## 6. 审查任务驱动闭环

主接口：

```text
POST /chemical/upload-review
```

输入：

- `review_task`
- `target_markets`
- `top_k`
- `sds_file`
- `formula_file`
- `process_file`

系统不会只用标题生成一个总 query，而是把用户审查任务拆成多个子任务：

- 资料完整性与补件判断
- 物料识别和 CAS/浓度核验
- 工艺适用性和温压/步骤风险
- 储运兼容性和禁忌组合
- CN/EU/US 法规初筛

每个子任务生成独立 RAG query，并记录在 `rag_queries`、`retrieval.by_query` 和 `agent_branches` 中。

## 7. 多 Agent 分工

### 资料完整性 Agent

核对 SDS、配方表、工艺说明是否足以支撑预审。缺 SDS 章节、修订日期、供应商、CAS、浓度、工艺温压、关键步骤、储存条件时，生成补件动作并触发复核。

### 物料 Agent

识别物质、CAS/EC、浓度、别名和主数据命中情况。未知 CAS、缺 CAS、保密成分缺少合规声明时进入复核。

### 工艺 Agent

检查温度、压力、关键步骤、设备和混配关系。可燃液体与氧化剂同釜混配等情形触发硬性拦截。

### 储运 Agent

检查储存类别、禁忌物隔离、UN 编号、运输信息和储运兼容性。储存条件缺失或禁忌组合进入复核或不合规。

### 法规 Agent

结合目标市场和 RAG 证据，进行危险化学品目录、SVHC、TSCA、OSHA HCS 等初筛。知识库未加载或未召回版本化证据时，不允许自动放行。

### 主审 Agent

按规则优先原则汇总各分支：

- `合规`：资料完整、知识库已加载、证据充分、未命中红线。
- `复核`：资料缺失、未知物质、知识库未加载、证据不足或分支冲突。
- `不合规`：命中硬性拦截规则，例如禁忌组合或企业红线。

## 8. 资料完整性与补件判断

这一模块不是展示字段抽取结果，而是决定审查是否具备继续判断的资料基础。输出包括：

- 完整性评分
- 阻断性缺口
- 字段级清单
- 原文证据
- 影响范围
- 补件动作
- 是否触发人工复核

这些结果进入 `review_workbench.document_quality`、`review_workbench.supplement_actions` 和 `chief_synthesis.review_items`。

## 9. 可审计技术证据

前端保留技术细节折叠区，用于证明链路真实执行：

- 知识 Chunk 和 vector_status
- 高频 query 和查询历史
- 多 query RAG 召回
- TopK 原始召回
- Rerank 后排序和原因
- 命中规则
- Agent 分支分析
- 主审汇总
- Trace JSON

## 10. 边界声明

当前 Demo 不做真实 OCR、完整法规覆盖、生产权限、多租户和企业系统集成。官方来源文本用于演示级可检索知识文档构建，真实业务上线前需要合法授权、持续更新、人工审核和客户 EHS/法规团队确认。
